# Design: agent_mailbox v1 (inter-agent work-requests)

Generated by /office-hours (lean) on 2026-06-28 ¬∑ Status: DRAFT ¬∑ Mode: internal infra
Repo: braisntext/hermes-sandbox ¬∑ Branch: claude/seo-rule-backfill

## Problem
Agents work in silos. The onsite-SEO agent can SEE work that belongs to another agent
(a thin page that needs more content; two pages cannibalizing one keyword) but has no
way to hand it off. Goal: a profile-scoped way for one agent to file a work-request to
another, so they loop-engineer tasks ‚Äî all still terminating in the existing gates
(consumer opens a PR ‚Üí auditor; or direct push for agents that push).

## Premises (from office-hours)
- **P1 AGREED ‚Äî substrate = per-profile volume JSON**, not GitHub Issues. `agent-mailbox.json`
  at `/opt/data/profiles/<profile>/`, read/written with file tools. Matches ledger/
  site-state/seo-rules, cron-safe, no new GitHub-API auth. Issues (CEO-visibility) is a
  later enhancement, surfaced via Telegram.
- **P2 AGREED ‚Äî first handoff is additive** (SEO ‚Üí content agent expands a thin page),
  NOT cannibalization‚Üímerge (destructive; comes later once additive is proven).
- **P3 CONTESTED ‚Üí resolved** ‚Äî CEO rejected "consumer waits for its next scheduled run".
  Handoffs must be reactive. RESOLVED: feasible ‚Äî `tools/cronjob_tools.py` exposes a
  `cronjob` tool wrapping `cron/jobs.py:trigger_job` (the remediation loop already uses
  it). The emitter triggers the consumer's job ‚Üí it runs in seconds, not hours.
- **P4 AGREED ‚Äî loop/race guards mandatory in v1** (not deferrable).

## Chosen approach: C ‚Äî direct-trigger + guards helper
Emitter writes a request to `agent-mailbox.json` and triggers the consumer via the
`cronjob` tool; consumer reads its inbox on wake, does the work, PRs it, marks done.
The mandatory loop guards live in ONE committed, cron-safe helper script that BOTH
agents call ‚Äî not reimplemented per prompt (that's where the loop/clobber hazard would
spread). Rejected A (guards inline per prompt ‚Üí hazard spreads) and B (dispatcher cron ‚Üí
more than v1 needs; it's step 2, the Editor-in-Chief).

## v1 slice ‚Äî exactly one of everything
- **One emitter:** onsite-SEO agent.
- **One request type:** `expand_thin_content` (SEO flags a page under a word-count
  threshold from site-state ‚Üí asks the content agent to expand it).
- **One consumer:** the content-generating agent (SEE OPEN QUESTIONS ‚Äî likely the
  gap-hunter; needs confirmation).
- No dispatcher, no merge handoff, no second emitter. Those are later.

### Components
1. **`agent-mailbox.json`** (per-profile volume, outside the clone). Array of requests:
   ```json
   {
     "id": "<uuid>", "type": "expand_thin_content",
     "from": "onsite-seo", "to": "content",
     "payload": {"url": "...", "word_count": 180, "reason": "thin"},
     "hops": 1, "created": "ISO", "ttl_days": 7,
     "content_hash": "<sha of from+to+type+url>",
     "status": "open|claimed|done|expired",
     "claimed_by": null, "claimed_ts": null, "result_pr": null
   }
   ```
2. **`onsite-seo/mailbox.py`** (committed, stdlib-only, cron-safe ‚Äî invoked BY PATH like
   build_sitestate.py, never execute_code/python -c). Sub-commands the agents call:
   - `enqueue` ‚Äî append a request; REJECT if a non-expired request with the same
     `content_hash` exists (dedupe) or if `hops >= MAX_HOPS` (loop guard); stamp TTL.
   - `claim <id> <agent>` ‚Äî atomically set status=claimed + claimed_by (per-page lock:
     refuse if another open/claimed request holds the same url).
   - `inbox <agent>` ‚Äî list open requests addressed `to` this agent, skipping expired.
   - `done <id> <pr_url>` / `expire-sweep` ‚Äî close out / GC stale (past TTL).
   Atomic writes (tmp + os.replace), profile path via `--mailbox <path>`.
3. **Prompt additions:**
   - SEO (emitter): after building site-state, if a page is under the word-count
     threshold AND no open request exists for it ‚Üí `enqueue` it, then trigger the
     consumer's job via the `cronjob` tool. Bounded: ‚â§N enqueues/run.
   - Content agent (consumer): at run start, `inbox` ‚Üí `claim` one ‚Üí expand the page ‚Üí
     open a PR (auditor gates) ‚Üí `done <id> <pr_url>`. Increment `hops` only if it in
     turn emits a follow-up request.

### Loop & race guards (P4 ‚Äî in the helper, tested once)
- **max-hop:** `MAX_HOPS` (default 3) ‚Äî enqueue refuses beyond it. Kills A‚ÜíB‚ÜíA‚Üí‚Ä¶
- **dedupe:** `content_hash` (from+to+type+url) ‚Äî no duplicate open requests.
- **TTL:** `ttl_days` (default 7) ‚Äî `expire-sweep` GCs stale requests.
- **per-page lock:** `claim` refuses if another open/claimed request holds the same url
  ‚Üí two agents never edit the same file at once.

### Profile-agnostic
Mailbox path, agent names, and consumer job_id all come from args/profile config ‚Äî
nothing hardcoded biglobster. Activating for finview = same helper, its own
`agent-mailbox.json`, its own job_ids.

## Open questions (resolve before building ‚Äî "the assignment")
1. **Which agent is the consumer** ("the content agent") for biglobster, and its
   **job_id** (the emitter needs it to trigger). Candidate: the content gap-hunter.
2. **Does the SEO agent's toolset include the `cronjob` tool** (to trigger)? If not,
   enable it for that job, or fall back to a frequent dispatcher (Approach B).
3. **Does the consumer open PRs** (auditor-gated) ‚Äî yes for content (vs SEO's direct
   push). Confirm the consumer has gh/PR capability.
4. **Thin-content threshold** (word count) ‚Äî add `word_count` to site-state so the
   emitter has the signal cheaply.

## Build sequence
1. `mailbox.py` helper + unit-test the guards locally (dedupe, max-hop, TTL, lock).
2. Add `word_count` to `build_sitestate.py` output (cheap signal for the emitter).
3. Emitter prompt block (enqueue + trigger) ‚Äî behind a bounded ‚â§N/run cap.
4. Consumer prompt block (inbox ‚Üí claim ‚Üí expand ‚Üí PR ‚Üí done).
5. Deploy helper to volume (chown), dry-run one handoff end-to-end, verify via trace.

## Success criteria
A thin page flagged by SEO results in: one request in `agent-mailbox.json`, the content
agent triggered within seconds, a PR that expands the page, the request marked `done`
with the PR url ‚Äî and re-running produces NO duplicate request (dedupe) and never two
agents on the same file (lock).

## NOT in v1 (deferred)
Cannibalization‚Üímerge handoff (destructive); the dispatcher/Editor-in-Chief cron;
GitHub-Issues mirror for CEO-visibility; multiple emitters; gated‚Üíauto trust per
(profile, handoff_type).
